Milestone 1

DF1 normalization foundation

covers early DF1 extraction / QC / split standardization
Purpose
Convert DeepFashion1’s benchmark-fragmented raw files into a stable normalized substrate that every later registry, event table, alignment pass, and QC procedure can trust.
Why
DF1 was not a single coherent table but a collection of benchmark-specific formats, file conventions, label structures, and partition contracts; without normalization, every downstream module would have been forced to re-interpret raw DF1 differently, causing schema drift, duplicate logic, and untraceable errors.
Achievement
DF1 was transformed into a reproducible normalized root with benchmark-wise standardized tables, global join keys such as image-level identifiers, split normalization, and audit-friendly separated term/evidence/event outputs, establishing DF1 as a stable evidence source instead of a moving raw-data target.
What changed
The project moved from raw-file handling to a reusable DF1 substrate that could support SSOT term construction, common event conversion, provenance tracking, and later enriched evidence extraction without reopening raw benchmark logic each time.
Milestone 2

FP normalization foundation

covers FP extraction / normalization / audit-ready separation
Purpose
Bring Fashionpedia into the same FEL-normalized operational world as DF1 so that both datasets can later share registries, event contracts, and ontology alignment logic.
Why
Even though FP already had stronger ontology structure than DF1, it still needed to be converted into FEL’s normalized contract; otherwise DF1 and FP would remain incomparable not because of semantics, but because of incompatible data engineering assumptions.
Achievement
FP was normalized, QC’d, and separated into term/evidence/event-friendly outputs, which made FP a clean structured source for categories, parts, and attributes and gave the pipeline a second ontology-rich anchor rather than a single-dataset system.
What changed
The system stopped being a DF1-only pipeline and became a true cross-dataset ontology project with one fashion dataset optimized for broad raw coverage and another optimized for structured semantic annotation.
Milestone 3

Dataset-specific term SSOT fixed

DF1 registry + FP registry stabilization
Purpose
Freeze each dataset’s vocabulary into a stable, dataset-internal source of truth before attempting any cross-dataset semantic mapping.
Why
You cannot build trustworthy ontology alignment on top of shifting local vocabularies; first the project needed deterministic term registries for DF1 and FP so that every raw id, label, and variant could resolve into a stable term identity.
Achievement
DF1 and FP each received a fixed term registry covering their local vocabularies, creating stable term UIDs, stable join behavior, and a proper basis for QC, alignment generation, alias management, and later canonicalization into FEL Core.
What changed
Term logic moved from ad hoc parsing inside event builders to dedicated registry-based resolution, which sharply improved reproducibility and made semantic debugging possible at the vocabulary level instead of only at the row level.
Milestone 4

Common event schema established

DF1/FP common v2 + merge readiness
Purpose
Convert normalized dataset tables into a shared event contract so that DF1 and FP can be merged, queried, and aligned through one operational event language.
Why
Without a shared event schema, ontology alignment would still be trapped at the vocabulary layer; the project needed a unified event representation so that subjects, terms, partitions, labels, and evidence could be interpreted consistently across datasets.
Achievement
Both datasets were converted into common v2 events, DF1 split vocabulary was normalized, compatibility QC was added, and a merged FEL event table became possible, turning two independent sources into one mergeable semantic event space.
What changed
The pipeline crossed the boundary from “parallel dataset preparation” to “integrated multi-source semantic event processing,” which was the precondition for any real FEL-wide ontology work.
Milestone 5

Pairwise alignment was abandoned for Core Hub

strategic pivot from HITL-A to HITL-B
Purpose
Replace direct DF1↔FP pairwise term alignment with a hub-and-spoke strategy in which every source maps to a neutral FEL Core.
Why
Pairwise alignment looked workable for two datasets but would not scale to three, four, or more sources; it also risked making FP’s ontology structure silently become the FEL global standard, which would distort later expansion toward aesthetics, emotion, multilingual work, and additional fashion corpora.
Achievement
The project made its most important architectural decision: FP ceased to be the implicit destination ontology, and FEL Core became the real hub, changing the problem from repeated pairwise harmonization to one-time source-to-core convergence.
What changed
This pivot converted the system from a brittle alignment project into a scalable ontology platform where new datasets can be added by mapping into Core instead of re-opening every earlier alignment relationship.
Strategic consequence: this was the moment the project became a true ontology infrastructure rather than a two-dataset reconciliation exercise.
Milestone 6

FEL Core seed and selective HITL created

core seed + review queue + validated core alignment
Purpose
Generate a canonical core vocabulary, create source-to-core candidates, and use selective human review only where automation is risky rather than forcing full manual inspection of all rows.
Why
A fully manual pass over thousands of apparently exact candidates is wasteful and low-value; the project needed a mechanism that kept human oversight where ambiguity, noise, or semantic risk was high while letting straightforward cases scale automatically.
Achievement
FEL Core seed terms and source-to-core candidates were generated, review queues isolated the hard cases, human decisions were QC’d, and a validated source-to-core alignment layer was established, giving the project a real canonical vocabulary hub instead of a registry collection.
What changed
The ontology no longer depended on direct dataset vocabularies for meaning; instead, datasets began speaking through Core, enabling later alias consolidation, canonical event remapping, and ontology node stabilization.
Milestone 7

Facet taxonomy was fixed by human review

facet candidates → HITL → final facet completeness
Purpose
Assign every canonical core term a final semantic facet so that ontology typing is not probabilistic, partial, or split across multiple provisional files.
Why
Automatic facet classification alone was too risky for ambiguous attributes, nickname-like terms, garment details, and unresolved blanks; without a complete facet assignment, the ontology could not support stable facet edges, semantic grouping, or definition graph construction.
Achievement
Facet candidates were generated, high-impact and uncertain cases were reviewed by humans, unresolved blanks were repaired, and the final core term file reached facet completeness, meaning canonical terms were no longer semantically under-typed.
What changed
The project moved from “candidate semantic labeling” to a closed taxonomy assignment where terms could reliably participate in ontology edges, parent grouping, rule design, and later inference logic.
Milestone 8

Backbone ontology was materialized

canonical terms + facet edges + alias edges + graph export
Purpose
Turn the emerging ontology components into explicit node and edge tables that can drive graph export, search, and later ontology/evidence linkage.
Why
Before this point the project had ingredients—core terms, facets, aliases—but not yet a materialized ontology backbone; the system needed an actual graph-ready scaffold, not just intermediate CSVs.
Achievement
The backbone was materialized as canonical term nodes, term→facet edges, and source→canonical alias edges, with later graph export to Neo4j confirming that the ontology backbone could be loaded and traversed as a real graph rather than treated as loose tabular metadata.
What changed
Ontology became operational at the graph-structure level: users could inspect the canonical vocabulary backbone directly, and the project gained a stable intermediate representation from which more expressive ontology layers could later evolve.
Milestone 9

Ontology and evidence were redesigned as separate SSOT layers

Stage 40–42 conceptual re-foundation
Purpose
Reframe the project into distinct ontology, evidence, and inference layers so that graph semantics, observational data, and downstream reasoning are not conflated.
Why
The first joint SSOT attempt exposed structural mismatches: ontology edges were still too shallow, event contracts over-compressed subject/object structure, and feature anchoring failed against image tables; this proved that the system needed a formal architecture split, not just more patching.
Achievement
The project explicitly separated ontology from evidence and positioned inference above both, which clarified responsibilities: ontology would define meaning, evidence would preserve observational provenance, and inference would later consume both without rewriting their contracts.
What changed
This redesign prevented the project from collapsing ontology semantics into event engineering and became the conceptual basis for the later successful ontology final and evidence final layers.
Critical insight: the first SSOT v1 failure was not wasted work; it exposed the exact architectural fault lines that had to be split apart for the final system to become stable.
Milestone 10

Ontology final became an operational definition graph

final core terms / facets / aliases / definition edges
Purpose
Upgrade the ontology from a backbone-only graph into a definition-capable operational graph that encodes not only typing and aliasing, but also semantic relations among categories, parts, and attributes.
Why
A term registry plus facet assignment is still not a full ontology; the system required explicit definition edges such as hierarchical and compositional relations so that fashion concepts could be read as structured semantics rather than just tagged nodes.
Achievement
Final ontology node tables were materialized, aliases were finalized, and definition edges such as SUBCATEGORY_OF, HAS_PART, and HAS_ATTRIBUTE were added to IN_FACET and ALIAS_OF, converting the graph into a usable semantic definition structure rather than a thin registry backbone.
What changed
From this point onward, the ontology could support category–part–attribute reasoning, structured retrieval, and rule-based aesthetic/emotional interpretation because the graph now expressed relationships among concepts, not just labels attached to concepts.
Milestone 11

Evidence final achieved bridge-level integrity

images / instances / events / core bridge validated
Purpose
Construct the evidence layer as a structurally valid bridge among images, instances, events, and canonical ontology terms.
Why
An ontology without evidence is only a dictionary, and evidence without stable foreign-key bridges cannot support audit, retrieval, or inference; the pipeline needed a final integrity check proving that image-level, instance-level, and term-level observations were mutually consistent.
Achievement
Image tables were rebuilt with usable file paths, instance tables were constructed across DF1 and FP, event anchors were improved, and the bridge QC passed with zero FK misses across images, instances, events, and core terms, showing that the ontology and evidence layers finally closed together as one coherent system.
What changed
The project crossed from ontology engineering into deployable knowledge infrastructure: evidence could now point into ontology terms through validated bridges, making downstream inference scientifically defensible rather than merely heuristic.
Milestone 12

DF1 evidence refactor isolated the remaining bottleneck

Stage 62–74 refactor and diagnosis
Purpose
Refactor DF1 evidence generation beyond the old common-events dependency and determine why DF1 semantic support remained weak even after the ontology and bridge structures were largely stabilized.
Why
Although the system had reached bridge-level integrity, DF1 still underperformed in instance-level semantic support: support propagation was sparse, segmentation and item-attribute evidence were not sufficiently carrying canonical semantics forward, and later event anchoring quality depended on fixing this hidden weakness.
Achievement
The project introduced enriched DF1 sources, rebuilt DF1 extraction with raw + normalized + enriched evidence inputs, reconstructed images and instances under the refactor line, and finally isolated the true remaining bottleneck to three concrete areas: DF1 support-source weakness, term-alias coverage loss, and segmentation/image canonical UID alignment.
What changed
Instead of vaguely knowing that “DF1 support is weak,” the project ended with a sharply localized diagnosis, which is exactly what a mature pipeline needs before the next engineering cycle: the unsolved problem is now precise, bounded, and actionable.
Current frontier: the ontology and bridge infrastructure are largely in place; the main unresolved work is improving DF1-side semantic support propagation so that instance anchoring becomes richer without sacrificing precision.

Closing summary

In 12 milestones, the project can be described as: raw dataset normalization → dataset SSOT vocabularies → common event unification → core-centered ontology convergence → human-stabilized facet taxonomy → materialized backbone → ontology/evidence layer split → operational definition graph → evidence bridge validation → DF1 bottleneck localization.
This condensed version is intentionally achievement-centered. It omits run commands and file-path-heavy narration, and instead explains what each milestone made possible scientifically, structurally, and operationally.
