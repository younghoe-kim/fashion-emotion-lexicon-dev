Executive summary

The full project progressed from heterogeneous raw dataset normalization, through deterministic source-vocabulary freezing and common event unification, into a Core Hub–centered ontology pipeline with selective HITL governance, final ontology materialization, and a bridge-validated ontology–evidence backbone. The decisive endpoint is not just a cleaned ontology, but a completed semantic infrastructure capable of supporting traceable inference.

Milestone 1

DF1 raw normalization contract established

benchmark-fragmented raw source → normalized benchmark SSOT
Reason
DF1 did not exist as a single coherent table. It was a family of benchmark-specific raw files with different parsing assumptions, partition conventions, label structures, and geometry formats.
Core significance
This stage established the first real source-of-truth substrate for DF1 and prevented every later component from re-parsing raw data independently.
Detailed milestone fields

Purpose

Convert DeepFashion1 raw benchmark files into stable normalized benchmark tables that can be consumed by registries, common-event builders, QC routines, and evidence reconstruction without reopening raw benchmark logic.

Why

Without normalization, DF1 would remain a moving target. Each downstream script would need its own raw parser, which would create schema drift, duplicate logic, and untraceable inconsistency across stages.

Achievement

DF1 was transformed into a reproducible normalized root with benchmark-wise standardized outputs, canonical image-level join logic, split-aware tables, manifest/QC artifacts, and source files suitable for later registry and event building.

What changed

The project moved from ad hoc raw-file handling to a reusable DF1 source substrate. From this point on, later builders could consume normalized tables rather than benchmark-specific raw assets.

Algorithms

Benchmark-specific parser routing; canonical image_uid generation; benchmark-normalized table materialization; global merge logic for image-linked content; split-aware extraction across CAPB / consumer2shop / in-shop / landmark families.

Policy

Strict mode is allowed to fail hard when source assumptions break. Optional empty outputs are not materialized as meaningless blanks. Normalized outputs become the only downstream-authorized DF1 substrate.

QC

Manifest generation, benchmark-level sanity checks, split integrity checks, parser-level failures in strict mode, and normalized output consistency validation.

Outputs

benchmarks/*/normalized/*.csv.gz, DF1 normalized root, QC summary files, and manifests used by later registry and common-event stages.

Milestone 2

FP normalization contract established

Fashionpedia raw source → object-centric normalized substrate
Reason
Fashionpedia had stronger object semantics than DF1, but it still needed to be translated into FEL’s normalization contract to become mergeable, auditable, and ontology-ready.
Core significance
This stage gave the pipeline a second normalized source with strong instance semantics, which later became crucial for ontology edge induction and instance-aware evidence modeling.
Detailed milestone fields

Purpose

Convert Fashionpedia’s raw images, annotations, instance objects, and attributes into a normalized source system compatible with the FEL architecture.

Why

Even ontology-rich sources cannot be used directly. Stable IDs, provenance, sparse attributes, item materialization, and normalized indices are required before graph integration becomes safe.

Achievement

FP was normalized into image indices, instance tables, sparse attributes, item tables, and auxiliary outputs under a provenance-preserving and hard-fail capable normalization regime.

What changed

The project ceased to be DF1-only and became a true multi-source pipeline with one evidence-rich source and one object-ontology-rich source.

Algorithms

Normalized image index construction; global instance_uid assignment; sparse attribute extraction; item table construction; degenerate geometry detection and geometry flagging.

Policy

Required instance JSON may trigger hard failure if missing; raw instance IDs are not treated as global PKs; provenance is preserved as dataset-relative source references instead of opaque ETL internals.

QC

Image-index consistency, instance/category/attribute source completeness, sparse attribute referential checks, and hard-fail enforcement for missing required structural inputs.

Outputs

fp_images_index.csv, normalized instances, sparse attributes, item tables, and a normalized FP source root consumed by registry and common-event builders.

Milestone 3

DF1 deterministic term registry fixed

dataset-specific vocabulary frozen as a stable registry SSOT
Reason
Cross-dataset ontology alignment cannot be trusted unless each source vocabulary is frozen first. The source itself must stop moving before canonicalization begins.
Core significance
This stage created the first deterministic vocabulary backbone for DF1 and enabled all later term-level joins, mappings, and alias generation.
Detailed milestone fields

Purpose

Materialize DF1’s source vocabulary as a dataset-internal SSOT containing term identity, normalized names, term types, and controlled auxiliary semantics.

Why

Without a deterministic term registry, later builders would continue to re-infer term identity from raw labels and source files, leading to non-reproducible joins.

Achievement

DF1 category, attribute, color, and auxiliary-term information were frozen into a single registry with stable term UIDs and normalization-aware identity logic.

What changed

Term resolution moved out of individual extraction scripts and into an explicit vocabulary layer, making ontology alignment scientifically inspectable rather than implementation-implicit.

Algorithms

Normalized-name generation; source autodiscovery; deterministic term identity construction; optional dense vs physical term-id policies; auxiliary type normalization across DF1 term families.

Policy

CJK leakage is filtered; consumer2shop vocabulary follows source-language restrictions; duplicate normalized names are surfaced as warnings instead of being silently collapsed.

QC

Header validation, BOM leak checks, duplicate UID and duplicate normalized-name detection, source-family coverage checks, and registry consistency warnings.

Outputs

df1_term_registry.csv plus associated registry QC files.

Milestone 4

FP deterministic term registry fixed

Fashionpedia vocabulary frozen as a source-specific SSOT
Reason
Fashionpedia’s source vocabulary also had to be frozen before ontology-level convergence. Taxonomy presence alone was not enough; actual normalized IDs and references had to be captured.
Core significance
This stage ensured that FP’s categories, parts, and attributes could later participate in the Core Hub as stable source terms rather than raw taxonomy labels.
Detailed milestone fields

Purpose

Freeze FP’s vocabulary into a dataset-specific registry that fully captures the IDs and normalized semantic terms referenced by later event builders.

Why

A source taxonomy is not identical to a downstream-usable term registry. Event-layer IDs must be preserved rather than overwritten by incomplete or idealized vocabulary assumptions.

Achievement

FP gained a coverage-preserving registry with normalized terms, stable identities, and taxonomy-enriched semantics for category, part, and attribute families.

What changed

FP term identity became explicit and reusable, allowing later common-event conversion and source-to-core alignment to operate on stable source vocabulary objects.

Algorithms

Registry assembly from normalized outputs; taxonomy enrichment; term normalization; coverage retention over source-referenced IDs.

Policy

Normalized-output references are authoritative; taxonomy metadata enriches registry meaning but should not erase event-backed IDs or source coverage.

QC

Coverage checks across category/part/attribute IDs, duplicate handling, and normalized registry integrity validation.

Outputs

fp_term_registry.csv and corresponding QC artifacts.

Milestone 5

DF1 common event contract materialized

normalized DF1 observations translated into a shared event layer
Reason
Vocabulary SSOT alone cannot power a graph system. Observations also need to be moved into a common event language.
Core significance
This stage created the first real observational contract for DF1 and made later merge, alignment, audit, and evidence reasoning possible.
Detailed milestone fields

Purpose

Translate DF1 normalized benchmark outputs into a common event schema that can be merged, inspected, and later aligned to canonical ontology terms.

Why

Without a common event layer, the ontology pipeline would remain disconnected from source observations and would only operate at vocabulary level.

Achievement

DF1 event families such as category, attribute, bbox, landmark, partition, and retrieval-pair evidence were materialized into one shared event contract.

What changed

DF1 observations stopped being benchmark-specific tables and became event-addressable semantic evidence.

Algorithms

Normalized-benchmark readers; registry joins; provenance injection; landmark schema correction; partition and retrieval-pair event construction; split-aware event generation.

Policy

Only normalized DF1 outputs are permitted as builder inputs; output folders are not reused as source inputs; provenance remains attached at the event layer.

QC

Schema checks, event-type counts, non-empty mapped-field checks, registry join completeness, and benchmark-family output validation.

Outputs

fel_events_df1_common_v2.csv.gz and builder QC summaries.

Milestone 6

FP common event contract materialized

Fashionpedia observations translated into the same event family
Reason
Cross-dataset ontology work requires a shared observation contract, not just two separate vocabularies and two separate normalizations.
Core significance
This stage brought Fashionpedia into the same event-centered semantic space as DF1.
Detailed milestone fields

Purpose

Convert FP normalized object annotations into the same common event contract used across FEL’s observational layer.

Why

A common ontology hub is not meaningful unless both datasets emit observations under a shared semantic schema.

Achievement

FP category, part, attribute, bbox, and geometry observations were represented in the same common-event framework used by DF1.

What changed

FP became event-compatible with DF1 and therefore compatible with unified downstream evidence processing.

Algorithms

Image-index-backed split attribution; object-centric event materialization for category, part, attribute, bbox, and geometry observations.

Policy

The image index is the split/provenance authority; object semantics must remain explicit rather than being collapsed into image-only labels.

QC

Schema completeness, event-type coverage, split validity, and registry-backed source-term reference checks.

Outputs

fel_events_fp_common_v2.csv.gz and corresponding builder QC artifacts.

Milestone 7

DF1 split vocabulary normalized to a merge-safe contract

lossless normalization of split semantics after DF1 common events
Reason
Even after common-event conversion, split values still contained heterogeneous surface forms that could undermine strict merge and QC logic.
Core significance
This stage stabilized the split vocabulary without destroying original raw split semantics.
Detailed milestone fields

Purpose

Map DF1 split labels into a restricted, merge-safe vocabulary required by later common-schema compatibility checks.

Why

Minor split-string differences can silently break downstream assumptions even when the underlying event schema is already aligned.

Achievement

query/gallery forms were normalized into val, unexpected tokens were sent to unknown, and raw split strings were preserved in metadata for traceability.

What changed

DF1 events became strictly compatible with later cross-dataset merge and comparison stages while retaining raw split provenance.

Algorithms

Restricted split-vocabulary mapping; controlled fallback to unknown; raw value retention via metadata fields.

Policy

Normalization must be lossless with respect to provenance: mapped split values support merge safety, while raw split values remain recoverable.

QC

Allowed-vocabulary enforcement, before/after split counts, and integrity checks ensuring raw split values remain recoverable after normalization.

Outputs

fel_events_df1_common_v2.splitnorm.csv.gz and split-normalization QC.

Milestone 8

DF1–FP common-schema compatibility QC stabilized

strict merge-readiness sanity contract
Reason
Two event tables are not enough. They must be proven compatible under a strict merge-readiness contract before they can be safely unified.
Core significance
This stage converted “shared schema by intention” into “shared schema by validation.”
Detailed milestone fields

Purpose

Validate that DF1 common events and FP common events satisfy identical structural and semantic expectations before downstream merge and ontology alignment.

Why

Schema names can match while hidden assumptions still differ; compatibility must be checked, not assumed.

Achievement

The pipeline gained a formal sanity gate that detects incompatible headers, duplicate event IDs, invalid split vocabularies, and incomplete mapped events.

What changed

Cross-dataset merge ceased being a procedural convenience and became a validated contract boundary.

Algorithms

Exact header comparison; duplicate and cross-collision detection for event_uid; restricted split-vocabulary validation; required-field checks for mapped events.

Policy

Event tables are not mergeable merely because they have similar names; they must satisfy exact structural and semantic readiness checks.

QC

Header equality, event-UID uniqueness, split validity, mapped-field completeness, and strict failure on merge-unsafe conditions.

Outputs

Compatibility QC JSON certifying whether DF1 and FP common events are merge-ready.

Milestone 9

Pairwise alignment was abandoned for the Core Hub architecture

architectural pivot from DF1↔FP to source→FEL Core
Reason
Direct pairwise alignment was workable only in the short term. It would not scale to many datasets and risked making one dataset’s ontology structure the de facto global standard.
Core significance
This was the decisive architectural pivot that turned the project from a two-source reconciliation workflow into a scalable canonical knowledge-graph platform.
Detailed milestone fields

Purpose

Replace direct DF1↔FP mapping with a dataset-neutral hub-and-spoke model centered on FEL Core.

Why

As datasets increase, pairwise alignment complexity grows combinatorially and semantic authority becomes unstable.

Achievement

FEL Core became the canonical semantic destination for all later source alignment work.

What changed

New sources could now be added by mapping them into Core rather than reopening all historical dataset-to-dataset mappings.

Algorithms

This milestone is an architecture pivot rather than a numeric optimization; downstream algorithms were reorganized around canonical hub convergence.

Policy

No source dataset is permitted to silently become ontology gold. Canonical semantics must be owned by the hub, not by a single upstream source.

QC

Evaluated indirectly through later candidate-generation stability, review-queue efficiency, merged alignment coherence, and final ontology materialization.

Outputs

Core Hub alignment strategy and the design basis for source-to-core candidate generation.

Milestone 10

Core seed generation and source-to-core candidate policy established

deterministic canonical grouping with exact keys and fuzzy hints
Reason
Once Core Hub became the alignment target, the system needed a reproducible mechanism for generating canonical seed terms and candidate source→core mappings.
Core significance
This stage created the canonical vocabulary hub and the first structured candidate layer for selective HITL.
Detailed milestone fields

Purpose

Construct a canonical seed vocabulary and source-to-core candidate mapping space from DF1 and FP source registries.

Why

Selective review cannot begin without a structured candidate space and a deterministic notion of what counts as the same core concept.

Achievement

Canonical core terms, suggested source-to-core mappings, fuzzy hints, and source-provenance-bearing seed rows were generated.

What changed

Source registries began converging into a shared canonical vocabulary instead of remaining parallel and isolated.

Algorithms

String normalization; auxiliary-type normalization; exact grouping by (term_type, aux_type_norm, term_name_norm); deterministic core UID generation via SHA1-16; token-overlap fuzzy hints; shortest non-empty source term selected as a human-friendly display heuristic.

Policy

Exact canonical key determines true core grouping. Fuzzy similarity provides review hints only and never silently becomes final canonical authority. Provenance remains attached to every generated core seed.

QC

Counts by source and term type, core term totals, candidate totals, and problem logging for missing normalized values or missing type/name combinations.

Outputs

fel_core_seed_terms.v1.csv, source_to_core_candidates.v1.csv, and seed-generation QC JSON.

Milestone 11

HITL template contract established

candidate mappings translated into a reviewer-editable decision sheet
Reason
Human review had to operate on a controlled decision schema rather than raw candidate tables that mixed suggestion data with source metadata.
Core significance
This stage turned source-to-core candidate generation into an actual human-review interface.
Detailed milestone fields

Purpose

Create a stable, reviewer-editable alignment sheet that encodes one row per source term and exposes explicit decision fields.

Why

Unstructured human edits are hard to merge, hard to QC, and hard to reproduce. Review needs a contract, not just a spreadsheet dump.

Achievement

Candidates were turned into a standardized template with editable action, final core UID, display-name, reviewer, and note fields.

What changed

Review decisions became structured and mergeable rather than informal edits over raw candidate artifacts.

Algorithms

source_term_uid-based deduplication; suggested-core aware default action logic (MAP when a suggestion exists, KEEP otherwise); fixed output schema generation.

Policy

Every source term receives one reviewer-editable decision row. Reviewer intent must be expressed through explicit action fields, not through implicit edits to candidate metadata.

QC

Template schema integrity, row uniqueness by source term, and compatibility with downstream merge/QC scripts.

Outputs

hitl_core_alignment.template.v1.csv.

Milestone 12

Risk-scored review queue and selective HITL governance established

human attention focused on ambiguity, noise, and audit sampling
Reason
Most rows were too trivial to justify full manual review. The hard problem was identifying which rows truly deserved human attention.
Core significance
This stage made HITL scalable by turning it into a risk-aware review policy rather than a full-table manual pass.
Detailed milestone fields

Purpose

Prioritize human review where semantic ambiguity, noisy tokens, or candidate uncertainty are highest while avoiding redundant review of low-risk exact rows.

Why

Large-scale ontology alignment is bottlenecked not by the existence of candidates, but by misallocation of reviewer effort.

Achievement

The project established a selective review queue with explicit priority signals and a controlled random-audit pathway.

What changed

Human attention became targeted and auditable. Review ceased to be blanket inspection and became governance over semantic risk.

Algorithms

Priority scoring based on match_method != exact_core_key, low top similarity score, tied top candidates, non-ASCII content, weird tokens, empty normalized strings, too-short forms, digit-heavy patterns, plus controlled random audit sampling.

Policy

Low-risk exact rows are usually excluded from human review unless selected as audit samples. Ambiguity and lexical pathology increase review priority.

QC

Queue counts by reason, ambiguous/tied candidate statistics, audit sample counts, and reproducible queue-generation summaries.

Outputs

review_queue.v1.csv and queue QC JSON documenting why rows were selected.

Milestone 13

Deterministic auto-map generation established

automatic canonical mapping only under unique exact resolvability
Reason
Automation was useful, but only if its semantics were exact. Ambiguous suggestions had to remain ambiguous instead of being silently committed to ontology truth.
Core significance
This stage separated safe canonical automation from human-required semantic judgment.
Detailed milestone fields

Purpose

Automatically generate source-to-core mapping decisions only when exact, deterministic, unique canonical resolution is possible.

Why

Without an exactness boundary, auto-mapping becomes hidden ontology authorship rather than controlled assistance.

Achievement

Automatic MAP decisions became formally constrained to uniquely resolvable source terms under a canonical key.

What changed

Large subsets of source vocabulary could now be canonicalized automatically without blurring the line between deterministic resolution and semantic ambiguity.

Algorithms

Exact matching by (source_term_type, norm(source_aux_type), source_term_name_norm) against the Core seed; MAP emitted only if exactly one core term matches; zero-match and multi-match cases skipped explicitly.

Policy

Candidate suggestions are advisory. Final automatic mapping is recomputed from the canonical core seed and cannot rely blindly on prior fuzzy suggestions.

QC

Counts for auto-mapped rows, skipped no-core-match rows, and skipped ambiguous-core-match rows; deterministic-key uniqueness checks.

Outputs

Auto-generated MAP alignment file covering only uniquely resolvable source terms.

Milestone 14

Auto + human merged alignment QC stabilized

one-source-one-decision alignment under explicit human override
Reason
Auto outputs and human-reviewed outputs had to be merged into one final alignment layer without duplicate authority, delimiter drift, or semantic inconsistency.
Core significance
This stage created the final canonical alignment decision contract that later ontology materialization could actually trust.
Detailed milestone fields

Purpose

Merge automated alignment and human-edited review results into a single authoritative source-to-core decision layer.

Why

If auto decisions and human decisions coexist without a unique-key contract, ontology materialization receives contradictory semantic instructions.

Achievement

The project enforced one authoritative decision per source_term_uid, with human rows overriding auto rows and merged files validated through strict QC.

What changed

Alignment became finalizable and machine-consumable rather than remaining split across raw auto and edited review artifacts.

Algorithms

Merge key = source_term_uid; auto file loaded first; human file replaces matching source-term rows; final file emitted in QC-compatible delimiter/header format.

Policy

Human override has semantic priority. Every source term must appear exactly once in the merged alignment layer.

QC

Delimiter sniffing, action validity (MAP/KEEP/DROP), source registry existence, core-seed membership for MAP rows, term-type compatibility, aux-type compatibility, and duplicate-structure checks.

Outputs

Merged final core alignment CSV and merged-alignment QC JSON.

Milestone 15

Facet taxonomy finalized and ontology master consolidated

reviewed semantics merged into one final ontology staging master
Reason
Canonical alignment alone was insufficient. Ontology materialization required one source-prioritized master table and a finalized facet taxonomy with unresolved ambiguity closed or surfaced.
Core significance
This stage created the single semantic staging master from which final ontology tables could be materialized.
Detailed milestone fields

Purpose

Consolidate duplicate representatives, nickname repairs, garment-detail refinements, and reviewed alignment decisions into a final ontology master while also finalizing the facet taxonomy.

Why

Final ontology tables should not be built from scattered intermediate artifacts. They require one semantically governed master and one finalized facet registry.

Achievement

The project produced a consolidated ontology master and finalized the facet taxonomy used for later IN_FACET and parent-term construction.

What changed

Ontology construction gained a single staging authority rather than relying on a loose collection of HITL outputs and partial refinement tables.

Algorithms

Priority merge across duplicate groups, nickname source, garment-detail source, and HITL results; facet normalization mapping; extraction of final facet registry from the consolidated master.

Policy

Master consolidation is source-prioritized and normalization-aware. Unsafe or unresolved facet states are surfaced rather than silently accepted.

QC

Facet completeness checks, blank-facet elimination, final facet-level counts, and consolidated master integrity tracking.

Outputs

refactored_dictionary_master_final.csv, core_facets.final.csv, and master/facet QC summaries.

Milestone 16

Final ontology tables, operational ontology edges, and bridge-validated evidence backbone completed the system

final ontology materialization + unified evidence closure
Reason
The project could not be considered operational until ontology tables, alias bridge, definition edges, image tables, instance tables, and final evidence events closed together under bridge-level QC.
Core significance
This stage transformed FEL from an ontology engineering project into a usable ontology–evidence knowledge infrastructure.
Detailed milestone fields

Purpose

Materialize final ontology registries and connect them to a structurally valid evidence backbone suitable for retrieval, audit, and later inference.

Why

Ontology without evidence is only a semantic dictionary, while evidence without ontology linkage is only ETL. A usable system needs both layers and a verified bridge between them.

Achievement

The project reached a completed ontology–evidence backbone with final core terms, final facets, final alias bridge, operational ontology edges, rebuilt image tables, unified instances, final evidence events, and strict bridge QC.

What changed

FEL became a graph-ready semantic infrastructure rather than a collection of partial registries and disconnected extracted observations.

Algorithms

Schema-flexible core-term materialization from heterogeneous ontology master columns; alias remapping via representative canonical terms plus registry metadata backfill; support-thresholded edge generation for HAS_PART/HAS_ATTRIBUTE; final image-table reconstruction; unified DF1/FP instance construction; anchor-resolution logic in final evidence events.

Policy

Ontology and evidence remain distinct SSOT layers; ambiguous item/image conditions are preserved rather than forcibly collapsed; bridge correctness is prioritized over superficial completeness.

QC

Final ontology QC (blank_uid, blank_facet, duplicate UID checks), alias QC, edge support/QC, image/instance/event/core bridge FK validation, and strict denominator coverage accounting.

Outputs

core_terms.final.csv, core_facets.final.csv, term_alias.final.csv, ontology_edges.final.csv, images.final.v5.csv, instances.final.v9_1.csv, evidence_events.final.v7_1.csv, and final bridge-QC JSON.
